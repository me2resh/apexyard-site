Founder Onboarding · building an AWS portal

An AI engineering org that follows process.

You're about to stand up a real software-engineering organisation — product, architecture, engineering, security, design — except the team is AI, running under a strict, automated SDLC that keeps the work production-grade. You give direction in plain English; ApexYard turns it into specs, decisions, tickets, reviewed code, and shipped software. It's not "an AI that writes code" — it's an AI org that follows process.

The arc: install a few prerequisites → set up the framework → walk one idea from spark to kicked-off project (idea → validate → spec → decisions → its own repo → plan → build). Under an hour — and by the end you'll have a validated PRD, recorded-and-challenged decisions, a milestone plan, and real tickets, all before writing a line of portal code. You'll work in Claude Code, mostly by just talking (see §2).

00Before you start — prerequisites

Most of this is one Homebrew line. On a fresh Mac, install Homebrew first:

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

Then install everything ApexYard needs in one command:

brew install git gh jq node pipx

Finish setup — authorise GitHub and put pipx on your PATH (then restart your shell):

gh auth login        # authenticate once
pipx ensurepath      # then restart your shell

Two more, outside Homebrew:

  • Claude Code — the AI agent you drive ApexYard with. Install from claude.com/claude-code, sign in, and keep it current with claude update.
  • Docker (optional) — only if you later use container components: brew install --cask docker.

What each piece is for:

Sanity check — after installing, each of these should answer:

claude --version && gh auth status && jq --version && node -v && pipx --version

01Set up (≈10 min)

1.1 — Fork + clone the framework

Fork github.com/me2resh/apexyard to your account, then:

git clone git@github.com:<you>/apexyard.git
cd apexyard
claude            # start Claude Code in the repo

1.2 — Configure it

Inside Claude Code, run /setup — three quick exchanges (describe your stack → accept defaults → confirm). Done.

/setup

1.3 — Turn on the premium skills

You've been issued a fine-grained access token (a read-only GitHub PAT) so you can install the paid skills without repo access. Treat it like a password. Two steps: install the CLI, then run the setup wizard.

Step 1 — install the CLI. Run from anywhere. Paste the token at the hidden prompt; we export it as APEXYARD_DEPLOY_TOKEN so the wizard reuses it in Step 2 (no second prompt):

read -rs APEXYARD_DEPLOY_TOKEN; export APEXYARD_DEPLOY_TOKEN; echo
pipx install "git+https://${APEXYARD_DEPLOY_TOKEN}@github.com/me2resh/apexyard-premium.git"

Step 2 — run the wizard. Run from your apexyard fork (it detects your roots from there). The --confirm flag lets you pick the gated (🔒) skills like /roundtable and enables them for you:

cd ~/path/to/your-apexyard-fork
apexyard-premium setup --confirm

When it lists components, type the numbers for the set you want — recommended: apexyard-search, aws-diagram, apexyard-admin, apexyard-deploy, apexyard-pdf, apexyard-playbooks, apexyard-loops, roundtable. Then restart Claude Code so the new skills load.

Missed one? Install any single skill later from your fork — --confirm enables it if it's gated:

apexyard-premium install roundtable --confirm

Where the token comes from: whoever sent you this guide generated it (GitHub fine-grained PAT, read-only). Ask them to refresh it if it expires — you never need write access to any private repo.

1.4 — Open your dashboard (optional, recommended)

A browser view of your portfolio — including a read-only spend view so you can watch token cost as you go (nothing caps you; you just see it). The dashboard is password-protected; a unique password is generated for your install at ~/.config/apexyard/admin.env the first time it starts.

Start it — run from your apexyard fork:

export APEXYARD_OPS_ROOT="$PWD" APEXYARD_PORTFOLIO_ROOT="$PWD"
cd "$APEXYARD_PORTFOLIO_ROOT/apps/apexyard-admin" && npm run dev

Open it pre-authenticated — no password typing. Mint a one-time login token from your stored password and open straight in:

source ~/.config/apexyard/admin.env
cd ~/path/to/your-apexyard-portfolio/apps/apexyard-admin
open "http://127.0.0.1:3000/login?token=$(node scripts/mint-login-token.mjs)"

Prefer to type it? Copy the password to your clipboard, then open the login page:

source ~/.config/apexyard/admin.env; printf '%s' "$APEXYARD_ADMIN_PASSWORD" | pbcopy
open "http://127.0.0.1:3000"

One click instead? Add the launcher items in §1.5 below.

02Your golden path — idea → kicked-off project

Read this first — it's a conversation, not a checklist. You don't have to memorise commands or run them in order. Just talk to Claude in plain English — "I want to build a customer portal on AWS", "help me decide between Cognito and my own auth", "what should I do next?" — and the framework picks the right skill and tells you the next sensible step. The slash-commands below are the named shortcuts; type them directly, or just describe what you want. Skip, reorder, or revisit any step — this is the typical arc, not a script. When in doubt, ask: "what's the best next step?"

1Capture the idea

/idea A customer portal on AWS where our clients log in, see their projects, and download reports.

Logs the concept to your backlog before you over-invest in it.

2Pressure-test it cheaply

/validate-idea

Five questions that catch "this isn't worth building yet" in ten minutes.

3Write the spec

/write-spec customer portal

Produces a PRD: problem, users, acceptance criteria — the contract the rest of the work is measured against.

4Give the project its own repo (private by default)

Mental model: your apexyard fork is mission control — process, roles, decisions, tickets. Your product code does not live in it. The moment an idea is worth building, it gets its own repository — private by default (it's your IP):

gh repo create <you>/customer-portal --private --clone

Then register it so the framework governs it (so /status, /tasks, search, and the SDLC gates see it):

  • Brand-new repo → add an entry (name + repo) to apexyard.projects.yaml.
  • Already has code → run /handover <owner/repo> — it scores the codebase and wires it in.

Rule of thumb: one product = one private repo; apexyard governs many from your one fork. The portal's tickets, PRs, decisions, and diagrams live in the portal's repo — not your apexyard fork.

5Make your technical decisions — and challenge them

Every real architecture call gets recorded and can be challenged before you commit:

/decide Cognito vs. self-managed auth for the portal

/decide records an AgDR (Agent Decision Record) — options, the choice, the trade-off. And it automatically offers to stress-test the call: right after recording, it invites Naqid, The Contrarian to challenge it. You can also run it yourself anytime:

/challenge Cognito vs self-managed auth

Naqid steelmans your choice, then attacks it: hidden assumptions, failure modes, cheaper alternatives, and a proceed / proceed-with-changes / reconsider verdict. Advisory only — it informs, it never blocks. For the big forks (monolith vs microservices, RDS vs DynamoDB), convene the whole room:

/roundtable Should the portal be a monolith or microservices on AWS?

Several role agents (Head of Eng, Solution Architect, Tech Lead…) debate it and converge on one written recommendation with an explicit "what we are NOT doing".

6Plan the work

/plan-initiative customer portal

Turns the spec into milestones → tasks, in dependency order.

7File the tickets

/feature Client login with Cognito
/task    Set up the AWS infra (VPC, RDS, ECS)

Real GitHub issues with acceptance criteria. (One ticket at a time — the system enforces it.)

8See the architecture

/c4              # system + container diagrams
/dfd             # data-flow + trust boundaries
/aws-diagram     # AWS architecture from your IaC (once you have Terraform/CDK)

9Build, under governance

/start-ticket <number>     # declare what you're working on
# ...write code with Claude...
# open a PR → Rex (the AI code reviewer) reviews every commit
# you give the explicit go → it merges

03The guardrails you'll feel

ApexYard is deliberately strict — that's what keeps AI output shippable:

  • Ticket-first. You can't edit code until a ticket is active (/start-ticket). Keeps work traceable.
  • Every PR gets reviewed by Rex before it can merge.
  • Merges need your explicit per-PR "approved". A plan-level "go" is never enough — the system stops and asks before merging. Merges are the one truly hard-to-undo step.
  • No direct pushes to main, no committed secrets, tests required. The hooks catch these for you.

If something blocks you and you don't know why, just ask: "why did that get blocked?" — the system explains which rule fired.

04A good first session

/idea            → capture the portal concept
/validate-idea   → is it worth building now?
/write-spec      → the PRD
gh repo create … --private   → the portal's own private repo + register it
/decide          → Cognito vs self-managed auth   (auto-challenged)
/roundtable      → monolith vs microservices       (the big one)
/plan-initiative → milestones + tasks
/feature         → file the first ticket (in the portal's repo)
/start-ticket 1  → begin
/c4              → see the shape of what you're building

By the end: a logged idea, a validated PRD, its own private repo registered with the portfolio, two recorded + challenged decisions, a milestone plan, real tickets, and your first diagram — all before a line of portal code. That's the point.
